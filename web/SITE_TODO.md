## Diagrams

- **Mermaid diagrams look bad in dark mode.** They render fine on the light (paper)
  theme but poorly on dark. Cause: we render each diagram to a **static SVG at build**
  (`rehype-mermaid`, `strategy: "img-svg"`, `mermaidConfig: { theme: "neutral" }` in
  `astro.config.mjs`) — a single baked image with one theme, served as an isolated
  `<img>` that can't respond to the site's dark/light toggle. "neutral" suits the light
  background only.

  Options to weigh when we tackle it (none done yet):
  1. **Render light + dark variants** at build and CSS-swap by `:root[data-theme]`
     (show/hide). Robust; doubles diagram assets. Likely the right answer.
  2. **Switch to `strategy: "inline-svg"`** and theme via CSS custom properties /
     `currentColor` so the SVG adapts live — but Mermaid's inline `<style>` complicates
     the strict CSP (we chose `img-svg` partly to keep `img-src 'self' data:` clean), so
     this needs a CSP rework.
  3. **Custom Mermaid theme** with a transparent background and stroke/text colours that
     read on both themes (fiddly; "neutral" doesn't).
  4. Revisit **Excalidraw** for bespoke diagrams (decided against earlier for
     source-of-truth/diffability reasons — see the brainstorming history).

  Affected content: `diagrams/FitSD — Diagrams.md` and the inline flowcharts (e.g.
  Adoption & Positioning). Rendering is gated behind `RENDER_MERMAID=1` (Docker build
  stage); local builds show code blocks.

## Security / CSP

- **Strict CSP blocks runtime inline styles site-wide (pre-existing).** The `style-src`
  generated by Astro (`security.csp`) is `'self'` + per-build hashes. Starlight and
  Expressive Code emit inline `style="…"` attributes (≈77 on a docs page) and set inline
  styles at runtime, which the hashes don't cover — the browser console shows dozens of CSP
  violations per page (≈19 home / ≈79 a docs page / ≈28 a form). Visually the site looks
  fine and core nav works, so impact is mostly console noise + minor blocked enhancement
  styling — but it's not a clean security posture.

  This is **not** caused by the interactive forms (same counts on form-free pages; the form
  logic uses CSS classes, not inline styles). It predates them.

  The wrinkle: Astro always adds style hashes, and CSP **ignores `'unsafe-inline'` when
  hashes are present** — so you can't just add `'unsafe-inline'`. Options:
  1. Deliver the CSP from **nginx** instead of Astro's `<meta>`, with `style-src 'self'
     'unsafe-inline'`. Keeps strict-ish scripts only if Astro's inline scripts are handled
     (they need hashes or `'unsafe-inline'`) — i.e. likely also `script-src 'self'
     'unsafe-inline'`, a real posture decision.
  2. Accept the violations (static site, no user input, near-zero XSS surface) and silence
     by documenting the rationale.
  3. Eliminate inline styles upstream (not feasible — they come from Starlight / EC internals).

  Needs a deliberate call on the security posture (it touches the syscert-style strict-CSP
  goal). Flagged from the interactive-forms work, where it surfaced.
